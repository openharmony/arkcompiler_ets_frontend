
# Target Build

This document describes the implementation of target build (multi-target products) in the ArkTS build system.

## Functionality Intro

### Source Roots Configuration

Multiple targets can be defined for each module in `build-profile.json5`. Each target specifies its own `sourceRoots`:

```json5
{
  "targets": [
    {
      "name": "official",
      "source": {
        "sourceRoots": ["./src/official_pages"]
      }
    },
    {
      "name": "test",
      "source": {
        "sourceRoots": ["./src/test_pages"]
      }
    }
  ]
}
```

During the build, `sourceRoots` is passed to the build system via `DependencyModuleConfig.sourceRoots` and stored in the `ModuleInfo.sourceRoots` field.

### Source Roots Usage

With `sourceRoots` configured, developers can import modules using a simplified path that omits the source root prefix:

```typescript
// Instead of: import { getName } from 'entry/src/official_pages/VersionInfo'
// Simplified:
import { getName } from 'entry/VersionInfo'
```

The build system resolves the simplified import path by trying each source root in priority order. This allows different targets to provide different implementations of the same module path without changing the import statements.

## Build System Impl

### Static Build (Path Section)

Method: `handleAddSourceRootPath` in `generate_arktsconfig.ts`

For the module's own source files, the build system generates a `paths` mapping so the compiler can resolve simplified imports.

**Algorithm:**

1. Reverse the `sourceRoots` array (hvigor sends them in reversed priority order, so reversing restores highest-priority-first).
2. For each source root, resolve its absolute path via `moduleRootPath`.
3. Append the module root path (`./`) as the lowest-priority fallback.
4. Register the mapping: `packageName → [highPriorityRoot, ..., lowPriorityRoot, moduleRoot]`.


**Generated example:**

```json
{
  "paths": {
    "entry": [
      "/project/entry/src/official_pages",
      "/project/entry/src/main",
      "/project/entry"
    ],
  }
}
```

### Interop Build (Dependency Section)

Methods: `addSingleModuleDependencies`, `addTransformedDependencies`, `addRemainingDependencies` in `generate_arktsconfig.ts`

For interop dependencies, the build system reads each dependency module's `decl-info.json` (generated by declgen) and registers dependency entries in `arktsconfig.json`.

#### Overview

The dependency resolution is split into two phases:

1. **Transformed dependencies** — Files matching a source root get a shortened key (source root prefix stripped).
2. **Remaining dependencies** — Files not matching any source root are registered with their full path as the key.

#### Transformed Dependencies (`addTransformedDependencies`)

This is the core algorithm that handles source root priority.

**Input:**
- `normalizedRoots`: Source roots normalized (backslashes → forward slashes, `./` stripped, filtered, **reversed** to highest-priority-first order)
- `files`: The `files` object from `decl-info.json`, keyed by relative file paths
- `claimedTransformedKeys`: A `Set<string>` tracking which transformed keys have been claimed

**Algorithm:**

```
for each sourceRoot:
    for each file in decl-info.json:
        if file does not start with sourceRoot prefix:
            skip
        
        transformedKey = packageName + "/" + file[len(sourceRoot):]
        
        if transformedKey already claimed:
            mark file as processed, skip
        
        claim transformedKey
        mark file as processed
        
        legacyKey = packageName + "/" + file
        register legacyKey → depItem
        
        if transformedKey ≠ legacyKey:
            register transformedKey → depItem
        
        if file is entry file:
            register packageName → depItem
```

**Key design decisions:**

| Decision                             | Rationale                                                                                                                                                                                              |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| First-claim-wins on `transformedKey` | When multiple files under different source roots produce the same transformed key (e.g., `entry/ets/Foo`), the highest-priority source root claims it. Lower-priority duplicates are skipped entirely. |
| Legacy key always uses full path     | Ensures backward compatibility with older hvigor versions that don't understand transformed keys.                                                                                                      |

**Example:**

Given `sourceRoots = ["src/target1", "src/target2", "src/target3"]` and files:
- `src/target1/ets/Foo` → `Foo1.d.ets`
- `src/target2/ets/Foo` → `Foo2.d.ets`
- `src/target3/ets/Foo` → `Foo3.d.ets`

Result:
| Key                       | Points to    | Reason                                                     |
| ------------------------- | ------------ | ---------------------------------------------------------- |
| `pkg/ets/Foo`             | `Foo1.d.ets` | Transformed key claimed by highest-priority root (target1) |
| `pkg/src/target1/ets/Foo` | `Foo1.d.ets` | Legacy key for the winning file                            |

Files under `target2` and `target3` are skipped because `pkg/ets/Foo` is already claimed.
